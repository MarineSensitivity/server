# h3t — OBIS biodiversity-by-hex tile service

Vendored from [CalCOFI/api-h3t-py](https://github.com/CalCOFI/api-h3t-py): the
**h3t tile factory**. Takes a base64-encoded SQL `SELECT` against a read-only
[DuckDB](https://duckdb.org) file and returns
[h3j](https://github.com/INSPIDE/h3j-h3t)-format JSON tiles for the
`h3tiles://` MapLibre protocol (R client: `mapgl::add_h3t_source()`).

On the MST server this serves **OBIS biodiversity indicators** (ES50, Shannon,
species richness, # records) summarized to H3 hexagons — the "biodiversity by
hex (OBIS)" leg of the ocean-indicators manuscript (§4.4). The authoritative
store is built by
[`obisindicators`](https://github.com/marinebon/obisindicators)
(`build_obis_h3_duckdb()`) from `s3://obis-open-data/occurrence/*.parquet` and
lives at `/share/data/obis/obis_h3.duckdb`.

## Store schema (registered as `obis`)

- `idx_h3(res, cell_id, n, sp, shannon, simpson, es)` — precomputed all-taxa
  indicators, res 1–7 (fast default layers; no per-request `lgamma`).
- `idx_h3_taxon(rank, taxon, res, cell_id, n, sp, shannon, simpson, es)` —
  precomputed per-taxon indicators for ranks phylum/class/order, clustered by
  `(rank, taxon, res)`. A single-taxon map reads this instead of aggregating
  `occ_h3` live, so it's as fast as the all-taxa layer.
- `occ_h3(res, cell_id, aphiaid, phylum, class, "order", family, genus,
  species, date_year, records)` — species-level store at resolution tiers
  3/5/7 for live taxon/year-filtered queries (ES50 computed on the fly via
  DuckDB `lgamma`). Clustered by `(res, taxonomy)` so a taxon filter prunes to
  a few row groups.

## The SQL contract

User `SELECT` must project **exactly** `cell_id` (BIGINT H3 index), `value`
(numeric), and optional `n`. `{{res}}` is substituted with the tile's H3
resolution (store tops out at res 7 → clamp with `LEAST({{res}}, 7)`).
See `app/sql_validate.py` for the full validator. The query templates are
generated by `obisindicators::obis_h3t_sql()`.

```bash
SQL='SELECT cell_id, es AS value, n FROM idx_h3 WHERE res = LEAST({{res}},7)'
Q=$(printf '%s' "$SQL" | base64 | tr -d '\n')
curl -s   "https://h3t.marinesensitivity.org/h3t/health" | jq .
curl -s   "https://h3t.marinesensitivity.org/h3t/stats?q=$Q&res_h3=4" | jq .
curl -s   "https://h3t.marinesensitivity.org/h3t/4/8/6.h3t?q=$Q" | jq '.cells[0:3]'
```

## Deployment (docker compose, on the MST server)

Wired into `../docker-compose.yml` as the `h3t` (FastAPI, :8889) + `h3tcache`
(Varnish, :6083) services and routed in `../caddy/Caddyfile`
(`h3t.marinesensitivity.org`). The `h3` + `spatial` + `httpfs` DuckDB
extensions are baked into the image (see `Dockerfile`).

```bash
docker compose up -d --build h3t h3tcache
docker compose logs -f h3t
```

### Release / cache invalidation

Build to a versioned file, symlink, restart, ban the cache; clients pass
`&release=vYYYYMMDD`:

```bash
ln -sf /share/data/obis/obis_h3_vYYYYMMDD.duckdb /share/data/obis/obis_h3.duckdb
docker compose restart h3t
docker compose exec h3tcache varnishadm 'ban req.url ~ "^/h3t/"'
```

## Configuration

See upstream env vars (`H3T_DBS`, `H3T_DEFAULT_DB`, `H3T_MAX_ROWS`,
`H3T_STMT_TIMEOUT_MS`, `H3T_CORS_ORIGIN`, `H3T_APP_GZIP`). On the MST server:

```
H3T_DBS=obis:/share/data/obis/obis_h3.duckdb
H3T_DEFAULT_DB=obis
```
